# Data Flow Architecture

## Overview

The CNC Digital Twin platform follows an event-driven, layered architecture where telemetry generated by CNC machines flows through multiple processing stages before being visualized on the frontend dashboard.

The data flow ensures:

- Reliable data ingestion
- Real-time dashboard updates
- Historical data persistence
- Alert generation
- Analytics computation
- Scalable processing

# Stage 1 — CNC Machine

The CNC machine continuously generates operational data from multiple sensors.

Typical sensor values include:

- RPM
- Temperature
- Vibration
- Power Consumption
- Tool Wear
- Feed Rate
- Spindle Load
- Machine Status

Example

```json
{
  "machineId": "CNC-001",
  "rpm": 3400,
  "temperature": 68.2,
  "vibration": 0.02,
  "power": 15.1,
  "status": "RUNNING"
}
```

---

# Stage 2 — Data Ingestion

The backend receives telemetry through REST endpoints (or MQTT in future versions).

```
POST /api/v1/telemetry
```

Responsibilities:

- Accept incoming telemetry
- Parse JSON payload
- Validate schema
- Authenticate source (if applicable)

---

# Stage 3 — Validation

Incoming data is validated before processing.

Validation includes:

- Required fields
- Numeric ranges
- Timestamp format
- Machine existence
- Data type validation

Invalid requests are rejected with appropriate HTTP status codes.

---

# Stage 4 — Business Logic

The service layer processes the validated telemetry.

Tasks include:

- Unit conversions
- Derived metric calculation
- Threshold evaluation
- Machine status updates
- Event generation

---

# Stage 5 — Database Persistence

Validated telemetry is stored in MongoDB.

```
Telemetry Service

↓

Telemetry Collection

↓

Indexed by

Machine ID

Timestamp
```

Benefits:

- Historical reporting
- Trend analysis
- Replay capability
- Predictive maintenance

---

# Stage 6 — Alert Processing

The Alert Service evaluates telemetry against configured thresholds.

Example Rules

| Metric | Threshold |
|---------|-----------|
| Temperature | > 80°C |
| RPM | > 5000 |
| Vibration | > 0.05 g |
| Tool Wear | > 90% |

If a threshold is exceeded:

```
Telemetry

↓

Threshold Check

↓

Alert Created

↓

Stored in Database

↓

Socket Notification
```

---

# Stage 7 — Analytics Processing

Analytics are generated from historical and real-time telemetry.

Calculated metrics include:

- OEE
- Availability
- Performance
- Quality
- Utilization
- Downtime
- Average Temperature
- Energy Consumption

---

# Stage 8 — Real-Time Broadcasting

After processing, the backend broadcasts updates through Socket.IO.

Events include:

```
telemetry_update

machine_status

dashboard_update

alert_created

analytics_update
```

Machine-specific rooms ensure only subscribed clients receive updates.

---

# Stage 9 — Frontend Processing

The frontend listens for socket events.

Flow

```
Socket Event

↓

Redux Action

↓

State Updated

↓

React Re-render
```

Only components affected by the new state are re-rendered.

---

# Stage 10 — Dashboard Visualization

Users immediately see:

- Updated telemetry charts
- Machine status
- Alert notifications
- KPI cards
- Analytics graphs
- Digital Twin state

No manual refresh is required.

# Interview Talking Points

When explaining the project:

- Describe the journey of telemetry from the CNC machine to the UI.
- Explain why MongoDB stores historical data while Socket.IO delivers live updates.
- Highlight the separation of concerns between controllers, services, and persistence.
- Explain where authentication, validation, analytics, and alert generation occur.
- Mention how the architecture can scale horizontally with Redis and Kafka.

---

# Summary

The CNC Digital Twin platform processes telemetry through a layered, event-driven pipeline. Data flows from CNC machines into the backend, where it is validated, persisted, analyzed, and broadcast to connected clients. REST APIs support resource management, while Socket.IO provides real-time updates, resulting in a responsive, scalable, and maintainable industrial monitoring system.
